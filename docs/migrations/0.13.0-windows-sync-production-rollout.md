# 0.13.0: Windows sync production rollout

Version 0.13.0 adds a supported per-user operating workflow around the V1 Windows read-only client.
Install a trusted local wheelhouse with `medlearn sync install-windows --wheel ...`. The client lives
under `%LOCALAPPDATA%\MedLearn\sync-client\venv`; token-free configuration and rollout state remain
under `%LOCALAPPDATA%\MedLearn\sync`, outside the Obsidian Vault.

Existing 0.12.0 configurations must run `medlearn sync pull --dry-run --json` before the next real
pull. The first real pull must include `--confirm-first-pull`. This migration does not move Vault
content, alter Worker or R2 contracts, upload local files, delete artifacts, or overwrite conflicts.

Preview the optional `MedLearn Vault Sync` task with `medlearn sync schedule install --what-if --json`.
Registration requires a successful first real pull. See `docs/windows-sync.md` for recovery, iCloud,
upgrade, uninstall, and first-production-Vault commands.

## Scheduled Task hardening

### Command-line quoting

The scheduled task action arguments are now serialized using the standard Windows
`subprocess.list2cmdline` convention rather than PowerShell single-quote escaping.
This correctly handles wrapper paths containing spaces, Chinese characters,
apostrophes, parentheses, and ampersands. PowerShell source-code quoting
(`_powershell_quote`) remains in use for its intended purpose — embedding Python
values inside generated `.ps1` source — but is no longer the Windows argv
serialization mechanism.

### Structured task status

`medlearn sync schedule status --json` now uses locale-independent PowerShell
ScheduledTasks cmdlets (`Get-ScheduledTask`, `Get-ScheduledTaskInfo`) with JSON
serialization instead of `schtasks.exe /Query` return-code inspection. The
structured result includes `state`, `last_run_time`, `next_run_time`,
`last_task_result`, `executable`, `arguments`, `wrapper_path`, and
`interval_minutes` where available. A missing task returns
`{"task_name": "MedLearn Vault Sync", "registered": false}`. Inspection failures
produce a stable `SYNC_SCHEDULE_FAILURE` error.

### Safe task removal

Task removal is now explicit and idempotent:

1. Check existence via PowerShell.
2. If absent, return `{"status": "already_absent", ...}`.
3. If present, unregister the exact task.
4. Verify absence.
5. Only then delete local metadata.
6. On failure, preserve local metadata and raise `SYNC_SCHEDULE_FAILURE`.

Removal no longer accepts `schtasks.exe` return code 1 as success (which could
mask access-denied, malformed invocation, or service failures). It never reports
`"removed"` when Task Scheduler still contains the task.

## Manual acceptance

CI validates task definition generation, argument serialization, and dry-run
registration but does not prove actual Task Scheduler execution. A Windows manual
acceptance gate is required before merge. Run the isolated acceptance script:

```powershell
# CI-safe validation (no task registration, no Worker contact):
powershell -NoProfile -NonInteractive -File scripts/acceptance/windows_sync_rollout.ps1 -ValidateOnly

# Full isolated acceptance:
powershell -NoProfile -NonInteractive -File scripts/acceptance/windows_sync_rollout.ps1 `
    -Endpoint "https://medlearn-cloud.<subdomain>.workers.dev" `
    -Wheel "dist/wheelhouse/medlearn_vault-0.13.0-py3-none-any.whl"
```

The script creates an isolated root with spaces and Chinese characters in its
path, uses a temporary `MEDLEARN_HOME` and throwaway Vault, never accepts the
sync token as a command-line parameter, and exercises the full lifecycle:
install → configure → login (interactive) → dry-run → first pull → schedule
install → structured status → task execution → token scan → removal → absence
verification → Vault integrity check.

See `docs/windows-sync.md` for the complete acceptance workflow and field-level
`schedule status` documentation.
