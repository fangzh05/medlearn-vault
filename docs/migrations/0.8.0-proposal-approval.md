# 0.8.0 proposal approval boundary

Package 0.8.0 adds `ProposalApprovalRecord` 0.1.0 without changing the persistent domain contract
version or the existing Proposal contract. Existing Proposal, Review, Execution, Job, intake, and
bundle objects require no migration.

Approval callers must supply the deterministic Proposal ID, the SHA-256 digest of the exact stored
Proposal object bytes, and the Proposal's expected base bundle digest. The service validates those
values against the fixed control store and creates only
`v1/approvals/<deterministic-approval-id>.json`. Consumers must validate the 0.1.0 schema and must
not treat an approval as permission for a different Proposal byte sequence or bundle state.

This release deliberately adds no commit endpoint, commit workflow, Vault credentials, Vault or
Obsidian writer, `LearningCapture` materialization, artifact upload, or LLM integration. A future
commit workflow must independently revalidate the immutable approval and its bound Proposal before
any write.
