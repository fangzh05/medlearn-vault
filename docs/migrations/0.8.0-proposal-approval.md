# 0.8.0 proposal approval boundary

Package 0.8.0 adds `ProposalApprovalRecord` 0.1.0 without changing the persistent domain contract
version or the existing Proposal contract. Existing Proposal, Review, Execution, Job, intake, and
bundle objects require no migration.

Approval callers must supply the deterministic Proposal ID, the SHA-256 digest of the exact stored
Proposal object bytes (`proposal_object_digest`), and the Proposal's expected base bundle digest.
This differs from `CaptureProposal.proposal_digest`, its internal semantic digest; both are checked.
The service validates those
values against the fixed control store and creates only
`v1/approvals/<deterministic-approval-id>.json`. Consumers must validate the 0.1.0 schema and must
also recompute `approval_id`, which JSON Schema cannot prove. One exact Proposal subject has one
immutable decision slot independent of decision: the first create-only decision wins, identical
decisions reuse it, and an opposite decision or changed rejection code is `APPROVAL_CONFLICT`.
Changing a decision requires a new Proposal or a future explicit supersession contract.

`decided_at` is the only approval timestamp. Rejections require a sanitized uppercase
`rejection_code`; approvals require it to be null. `source_job_id` is deliberately not stored
because approval does not verify Job or Execution provenance.

This release deliberately adds no commit endpoint, commit workflow, Vault credentials, Vault or
Obsidian writer, `LearningCapture` materialization, artifact upload, or LLM integration. A future
commit workflow must independently revalidate the immutable approval and its bound Proposal before
any write.
