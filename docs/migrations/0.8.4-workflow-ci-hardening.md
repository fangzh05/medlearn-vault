# 0.8.4 workflow and CI hardening

Propose and Approve are main-only workflows and always check out fixed `main` with credential
persistence disabled. `CONTROL_R2_*` credentials are step-scoped: checkout, setup, and dependency
installation do not receive them.

Approval now requires an explicit `approved` or `rejected` decision and an exact `proposal_id`
confirmation before its credential-bearing step. This is a workflow safety guard only: Approval
identity, canonical bytes, create-only behavior, reused semantics, rejection-code behavior, R2 key
layout, and all workflow and persistent contract versions are unchanged.

Repository CI pins Actions to immutable commits, installs a Python 3.12 hash-locked dependency set,
validates GERD and COPD bundles, and performs a local wheel smoke test. Worker CI uses its
repository-local pinned Wrangler for a deployment dry-run. No Vault publication capability was added.

Regenerate the CI lock with Python 3.12 and pip-tools 7.5.3:

```bash
python -m pip install pip-tools==7.5.3
python -m piptools compile --generate-hashes --resolver=backtracking --output-file requirements/ci.txt requirements/ci.in
```
